## 1. Goal & endpoint (locked)

Expose mpv's **gpu-next (libplacebo)** renderer through the **libmpv render
API**, which today only exposes the old `gpu` VO (mpv issue **#10810**). PR
**#16818** attempted it; maintainers rejected the approach (duplicated
`vo_gpu_next.c`, GL scattered in `gpu_next/context.c`).

- **Endpoint:** a *mergeable-quality* personal fork. Match the maintainer-spec
  architecture so it *could* upstream, but **success = a working fork you
  control**; upstream submission is optional, **not** gated on review/merge.
- **No-regression bar:** windowed `--vo=gpu-next` must stay
  **bit/behaviour-faithful** (≈ every mpv desktop user) + the Windows libmpv
  path you actually run. Full all-platform matrix only if you pursue upstream.
- **Approach:** slow, steady, manual, meticulous; behavior-preserving commits,
  golden-frame + perf verified after each.
